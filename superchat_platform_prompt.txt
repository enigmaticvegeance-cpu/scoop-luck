PROJECT OVERVIEW
--------------------------------------------------------------------------

Build a full-stack podcast superchat platform called "Scoop Luck" — a
YouTube Superchat alternative where viewers can tip a podcast/live-stream
crew directly, cutting out YouTube's revenue cut.

The platform has three user roles:

  1. ADMIN  — The podcast hosts (up to 4 people). Manage superchats,
              view donations, configure settings. Accessed via a
              separate /admin route with email + OTP login.

  2. VIEWER — Registered, logged-in fans. Must create an account to
              send superchats. Can set a display name + profile photo
              (400×400px with drag/zoom/crop tool). Their name and
              avatar appear on their superchat cards.

  3. GUEST  — Unauthenticated visitors. Can ONLY view the live superchat
              feed. Cannot send superchats. No login prompt shown unless
              they attempt to donate (then redirect to register/login).
              Guests have zero write access to any API endpoint.

TECH STACK
--------------------------------------------------------------------------

Frontend:
  - Next.js 14 (App Router, TypeScript, strict mode)
  - Tailwind CSS v3 + shadcn/ui for base components
  - Framer Motion (via MCP) for all animations & transitions
  - 21st.dev components (via MCP) for premium UI blocks
  - React Hook Form + Zod for all form validation
  - next-themes for dark/light mode (default: dark)

Backend:
  - Next.js API Routes (Edge Runtime where possible)
  - Prisma ORM with PostgreSQL (Supabase hosted)
  - Supabase Auth (email/password + OTP for admin MFA)
  - Upstash Redis for rate limiting, session cache, idempotency keys

Payments (India-first, international-ready):
  - Razorpay — primary gateway for Indian users
    · UPI collect + QR code flows (India only)
    · Domestic debit/credit cards (India-issued)
    · Net Banking + Wallets (Paytm, PhonePe, etc.)
    · Razorpay does NOT reliably support international cards;
      do NOT use it for international card payments
    · Webhook signature verification (server-side only)
    · Idempotency keys on every order creation
  - Stripe — international card gateway (Visa/Mastercard/Amex/Discover)
    · Used when user selects "International Card" option
    · Stripe Checkout or Stripe Elements (PCI-DSS delegated)
    · Stripe webhook verification with stripe.webhooks.constructEvent
    · Supports 135+ currencies; charge in USD, display equivalent INR
  - PayPal — secondary international gateway
    · For users who prefer PayPal wallet or PayPal-linked cards
    · PayPal JS SDK (client) + PayPal REST API (server)
    · Create order server-side, capture server-side after approval
    · Webhook: verify with PayPal SDK verifyWebhookSignature
  - NEVER store card numbers, CVV, or UPI PINs — delegate 100% to gateway

  Payment method selection UI:
    Detect user locale (from browser Accept-Language + IP via Cloudflare
    cf-ipcountry header) to show the most relevant options first:
      · Indian IP  → show: UPI (default) | Indian Card | International Card | PayPal
      · Other IP   → show: International Card (default) | PayPal | Indian Card (collapsed)
    User can always switch manually.

Infrastructure:
  - Vercel (frontend + API routes)
  - Supabase (Postgres + Auth + Storage for profile images)
  - Cloudflare (WAF, DDoS, Rate Limiting, Turnstile CAPTCHA, CDN)
  - Resend (transactional email — OTP, receipts, invoices)
  - Sentry (error monitoring, never log PII)
  - Upstash Redis (rate limiting middleware)

━━━ 2. PAYMENT FLOWS ━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

  UPI (Razorpay — India only):
    Step 1 — User enters amount + message → hits "Pay with UPI"
    Step 2 — Backend creates Razorpay order (server-side, signed)
    Step 3 — User sees: (a) UPI ID input field OR (b) QR code toggle
    Step 4 — Razorpay handles the collect/intent flow
    Step 5 — Webhook (POST /api/webhooks/razorpay) receives event;
             verify HMAC-SHA256 of razorpay_order_id|razorpay_payment_id
             with webhook secret
    Step 6 — On verified success: mark order paid in DB, assign tier,
             enqueue superchat for display, send invoice email
    Step 7 — On-screen invoice modal with download (PDF)
    NEVER trust frontend "payment success" callbacks alone.

  Indian Debit / Credit Card (Razorpay):
    - Use Razorpay hosted checkout or Razorpay.js Elements
    - Supports RuPay, domestic Visa/Mastercard issued in India
    - PCI-DSS delegated entirely to Razorpay
    - Same webhook verification flow as UPI

  International Debit / Credit Card (Stripe):
    - Stripe Elements or Stripe Checkout for card capture
    - Supports Visa, Mastercard, Amex, Discover, international cards
    - Charge in USD; show INR equivalent (at admin-configured rate)
    - Webhook (POST /api/webhooks/stripe): verify with
      stripe.webhooks.constructEvent(body, sig, STRIPE_WEBHOOK_SECRET)
    - On payment_intent.succeeded: mark paid, assign tier in INR-equiv,
      enqueue superchat, send invoice
    - Never trust client-side paymentIntent.status

  PayPal (International):
    - PayPal JS SDK (client) + PayPal REST API (server)
    - Create order server-side → client approves → capture server-side
    - Webhook: verify with PayPal SDK verifyWebhookSignature
    - Amount stored in USD, displayed as USD + approximate INR on card
    - Same tier logic as Stripe (INR equivalent at fixed rate)

  Common rules for ALL gateways:
    · Idempotency key = UUID v4 per order, stored in Redis TTL 24h
    · Never create duplicate orders for same idempotency key
    · Payment status: PENDING → PAID | FAILED | REFUNDED
    · Tier assigned server-side in webhook handler ONLY
    · Only PAID superchats appear in the live feed
    · Store: order_id, gateway, amount_paise_or_cents, currency,
             inr_equivalent_paise, tier, webhook_verified, created_at, paid_at
    · Never store: card_number, cvv, upi_pin, bank_password

  Invoice / Receipt:
    · Generated server-side with @react-pdf/renderer
    · Fields: invoice number, date, donor name (or "Anonymous"),
              amount (native currency + INR equivalent), gateway,
              transaction ID, platform name "Scoop Luck", logo
    · Emailed via Resend with PDF attachment
    · On-screen modal with "Download PDF" button
    · Invoice numbers: SL-INV-{YEAR}-{6-digit zero-padded sequence}

SECURITY REQUIREMENTS (NON-NEGOTIABLE)
--------------------------------------------------------------------------

Follow OWASP Top 10 and OWASP ASVS Level 2.

INPUT VALIDATION
  · Zod schemas on every API route input (server-side)
  · Allow-list validation (never block-list alone)
  · Strip all HTML from user text fields (DOMPurify server-side or
    he/entities library — never trust sanitize-html alone)
  · Validate content-type, request size limits on all routes
  · File uploads: validate MIME via magic bytes with file-type package +
    re-encode images with sharp; reject SVG entirely

INJECTION PREVENTION
  · Prisma parameterized queries only — never raw SQL string concat
  · No eval(), no Function(), no dangerouslySetInnerHTML with user data
  · Template injection: never pass user input to any template engine

AUTH & SESSIONS
  · Supabase Auth handles password hashing (bcrypt, never implement own)
  · JWT verification server-side on every protected request
  · Admin MFA: email OTP required on every login (not just first time)
  · CSRF: SameSite=Strict cookies + Cloudflare Turnstile on all forms
  · Session fixation: rotate session ID post-login

RATE LIMITING (Upstash Redis + Vercel middleware)
  · POST /api/payments/*          → 5 req/min per IP
  · POST /api/superchats          → 3 req/min per IP
  · POST /api/auth/login          → 10 req/15min per IP + email
  · POST /api/auth/otp            → 3 req/10min per email
  · GET  /api/admin/*             → 60 req/min per session
  · All other public endpoints    → 100 req/min per IP
  Return 429 with Retry-After header; log violations to Sentry

PAYMENT INTEGRITY
  · NEVER trust frontend payment status
  · Verify Razorpay webhook: HMAC-SHA256 of
    razorpay_order_id|razorpay_payment_id with webhook secret
  · Verify PayPal webhook with PayPal SDK
  · Idempotency: reject duplicate order creation for same key
  · Amount validation: re-verify amount in webhook matches order amount
    in DB — reject if mismatch
  · Webhook endpoints reject non-POST and wrong content-type immediately

SECURITY HEADERS (next.config.js)
  Content-Security-Policy: default-src 'self';
    script-src 'self' 'nonce-{nonce}' checkout.razorpay.com
               js.stripe.com www.paypal.com static.cloudflareinsights.com;
    style-src 'self' 'unsafe-inline' fonts.googleapis.com;
    img-src 'self' data: blob: *.supabase.co;
    connect-src 'self' api.razorpay.com api.stripe.com *.paypal.com
                *.supabase.co *.sentry.io;
    frame-src checkout.razorpay.com js.stripe.com www.paypal.com;
    font-src 'self' fonts.gstatic.com;
  Strict-Transport-Security: max-age=63072000; includeSubDomains; preload
  X-Frame-Options: DENY
  X-Content-Type-Options: nosniff
  Referrer-Policy: strict-origin-when-cross-origin
  Permissions-Policy: camera=(), microphone=(), geolocation=()

CLOUDFLARE (configure in dashboard)
  · WAF: OWASP Core Rule Set, India + international allow, block known
    bad bots and scanners
  · DDoS: Auto + rate limit rules matching API rate limits above
  · Turnstile CAPTCHA: on payment form, login form, registration form
  · Bot Fight Mode: ON
  · Security Level: Medium (High during live streams)
  · Cache: static assets only; API routes must bypass cache
  · Zero Trust: lock /api/admin/* to specific IP ranges (optional)

DATABASE SECURITY (Supabase / Postgres)
  · Row Level Security: enabled on ALL tables
  · Viewers: SELECT own rows only
  · Admins: all operations on all rows (via service role, server-side)
  · Public: SELECT on superchats WHERE status = 'PAID' AND hidden = false
  · Never expose service role key to frontend
  · Encrypt PII fields (email) at rest using pgcrypto if needed
  · Use UUIDs as primary keys everywhere

SECRETS MANAGEMENT
  · All secrets in Vercel environment variables (never in code/git)
  · Secret rotation plan documented in README.security.md
  · .env.example with placeholder values committed
  · .env* in .gitignore

LOGGING (Sentry + custom)
  Log:  auth attempts, admin actions, payment events, rate limit hits,
        webhook verifications, superchat hide/delete actions
  Never log: passwords, OTPs, card data, UPI PINs, full email addresses
             (log only hashed email for correlation), JWT tokens

ERROR HANDLING
  · Generic error messages to client ("Something went wrong")
  · Full error details only in Sentry
  · No stack traces, SQL errors, or internal paths in API responses
  · Fail closed: if auth check errors → deny access

DATABASE SCHEMA (Prisma)
--------------------------------------------------------------------------

model User {
  id            String    @id @default(uuid())
  email         String    @unique
  displayName   String?
  avatarUrl     String?
  role          Role      @default(VIEWER)
  createdAt     DateTime  @default(now())
  superchats    Superchat[]
}

model Superchat {
  id              String    @id @default(uuid())
  userId          String?   // null = anonymous
  user            User?     @relation(fields: [userId], references: [id])
  displayName     String    // captured at payment time
  avatarUrl       String?   // captured at payment time
  message         String
  amountPaise     Int       // in paise (INR) or cents (USD×100)
  currency        String    @default("INR")
  tier            Int       // 1–6
  gateway         Gateway
  gatewayOrderId  String    @unique
  gatewayPaymentId String?
  status          PaymentStatus @default(PENDING)
  webhookVerified Boolean   @default(false)
  hidden          Boolean   @default(false)
  invoiceNumber   String?   @unique
  createdAt       DateTime  @default(now())
  paidAt          DateTime?
}

model AdminSession {
  id          String   @id @default(uuid())
  adminEmail  String
  otpHash     String
  otpExpiry   DateTime
  verified    Boolean  @default(false)
  createdAt   DateTime @default(now())
}

enum Role { VIEWER ADMIN }
enum Gateway { RAZORPAY STRIPE PAYPAL }
enum PaymentStatus { PENDING PAID FAILED REFUNDED }
